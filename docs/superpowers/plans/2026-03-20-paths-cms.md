> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add standalone articles, content pillar taxonomy, and an editorial workflow to the LIMITLESS PATHS platform built on LearnHouse.

**Architecture:** Create Article and ContentPillar as new independent models (not modifying LearnHouse's core course/activity system). Reuse the existing TipTap editor for article content. Extend the RBAC Rights model with article and pillar permissions. Add editorial workflow state machine with 7 transitions. Build article dashboard in org UI and pillar management in admin panel.

**Tech Stack:** Python 3.13 / FastAPI / SQLModel / Alembic / PostgreSQL 16 / Redis 7.2 / Next.js 16 / React 19 / TypeScript / TipTap

- [ ] **Step 2: Extend Rights model with article and pillar permissions**

In `src/db/roles.py`, add a new permission model for articles:

```python
class ArticlePermissions(BaseModel):
    action_create: bool
    action_read: bool
    action_update: bool
    action_delete: bool
    action_submit_review: bool
    action_review: bool
    action_publish: bool

    def __getitem__(self, item):
        return getattr(self, item)
```

Add to `Rights` class (with defaults for backward compatibility):
```python
articles: ArticlePermissions = ArticlePermissions(
    action_create=False, action_read=True, action_update=False,
    action_delete=False, action_submit_review=False,
    action_review=False, action_publish=False,
)
pillars: Permission = Permission(
    action_create=False, action_read=True,
    action_update=False, action_delete=False,
)
```

Defaults are read-only â€” backward compatible with existing roles that don't have these fields in their stored JSON. LearnHouse's Pydantic model will use defaults when the JSON doesn't include these keys.

Note: `ArticlePermissions` is a custom class (not `PermissionsWithOwn`) because it has editorial-specific actions (`submit_review`, `review`, `publish`) that don't fit the standard CRUD pattern. The standard RBAC utility (`authorization_verify_based_on_roles`) only checks `action_{read|update|delete|create}`. For editorial workflow permissions, Task 6 will implement a custom permission checker that reads from the `ArticlePermissions` fields directly.

- [ ] **Step 1: Write workflow service**

Create `src/services/articles/workflow.py` with a state machine for article transitions:

```python
# Valid transitions: {current_status: {action: (target_status, required_permission)}}
TRANSITIONS = {
    "DRAFT": {"submit": ("IN_REVIEW", "articles.submit_review")},
    "IN_REVIEW": {
        "approve": ("APPROVED", "articles.review"),
        "reject": ("DRAFT", "articles.review"),
    },
    "APPROVED": {
        "publish": ("PUBLISHED", "articles.publish"),
        "revise": ("DRAFT", "articles.publish"),
    },
    "PUBLISHED": {"archive": ("ARCHIVED", "articles.publish")},
    "ARCHIVED": {"reopen": ("DRAFT", "articles.publish")},
}
```

Function: `transition_article(article_uuid, action, user, db_session, review_notes=None)`
- Validates current status allows the action
- Checks user has required permission
- For "submit": validates author is current user or user has articles.update
- For "reject": requires review_notes (400 if missing)
- For "approve": sets reviewer_id, review_date
- For "publish": sets published_at
- Creates a new ArticleVersion on every transition
- Returns updated article
